SECTION 1: PROVISIONING INTENT & LOGIC

In a managed cloud services environment, the AI agent acts as the intelligent orchestrator between user-expressed business intent and enforceable infrastructure state. The core competency lies in translating ambiguous or high-level requests into precise, secure, cost-aware, and compliant infrastructure configurations.

1.1 Intent-to-Decision Mapping Framework

| User Intent                  | Business Requirements                                | AI Agent Decision (Infrastructure Policy)                                                                 |
|------------------------------|------------------------------------------------------|-----------------------------------------------------------------------------------------------------------|
| High-Availability Production | Zero downtime SLA, SOC 2 / ISO 27001 compliance, audit-ready | • Multi-AZ EKS cluster with node groups across 3 AZs<br>• Multi-AZ RDS (PostgreSQL/MySQL)<br>• AWS WAF + Shield Advanced<br>• Strict IAM roles with least privilege<br>• Encrypted EBS volumes + KMS key rotation |

SECTION 3: NETWORKING & SECURITY DECISION TREES

Security is non-negotiable. The agent must apply zero-trust principles by default.

3.1 Public vs. Private Subnet Logic

Scenario: User requests: “Deploy a backend API for internal payroll processing.”

Agent Reasoning Chain:
1. Contains PII → regulated data
2. Internal use only → no external consumers
3. Therefore: must be in private subnet
4. Access pattern: via internal ALB or service mesh (e.g., Istio)
5. Never assign public IP
6. Ingress only from finance-app security group

Decision:
✅ Provision in private subnet
✅ Attach to internal ALB
❌ Block public IP assignment
❌ Deny internet gateway association

3.2 Secure Connectivity Patterns

For remote access (e.g., OpenVPN, Bastion):

- OpenVPN Deployment:
  - Deploy as container in private subnet
  - Expose only on UDP/1194 via NLB (not ALB)
  - Security Group:
    {
      "FromPort": 1194,
      "ToPort": 1194,
      "Protocol": "udp",
      "SourceCidr": "0.0.0.0/0",
      "Description": "OpenVPN entry point"
    }
  - But: VPC peering traffic must be restricted to 10.50.0.0/16 only
  - Enable MFA for admin users via SAML integration

Golden Rule: Public exposure must be minimal, monitored, and rotated.

SECTION 5: GUARDRAILS & FAILURE HANDLING

The agent must proactively prevent misconfigurations, overspending, and security violations.

5.1 Budget Enforcement Guardrail

- Constraint: Monthly budget = $500
- Request: “Provision a 10-node p3.16xlarge GPU cluster for ML training.”
- Agent Analysis:
  - p3.16xlarge ≈ $24.48/hr × 10 = $244.80/hr
  - Monthly (730h) ≈ $178,704 → 357× over budget
- Decision: REJECT
- Alternative Suggestion:
  > “Consider using g4dn.xlarge Spot instances ($0.526/hr) with checkpointing. Estimated cost: ~$385/mo. Would you like to proceed with this optimized configuration?”

5.2 Security Guardrail: Public S3 Bucket

- Request: “Create an S3 bucket for user uploads and make it public so the frontend can access files.”
- Risk: Public buckets are a top cause of data breaches
- Agent Decision: MODIFY + EDUCATE
- Action:
  1. Create private S3 bucket
  2. Attach bucket policy denying public access
  3. Suggest one of:
     - Option A: CloudFront + Origin Access Identity (OAI)
     - Option B: Presigned URLs (valid for 15 mins) generated by backend
     - Option C: Cognito-authenticated access via Amplify
- Response Template:
  > “Public S3 buckets are prohibited by security policy. Instead, I’ve configured a private bucket with secure access via [recommended method]. This prevents accidental data exposure while maintaining functionality.”

| Lift and Shift | On-prem → RDS | • VPN/Direct Connect setup<br>• Initial full load via DMS<br>• CDC for ongoing sync<br>• Cutover with minimal downtime |
| Sharding | Single DB → Distributed | • ProxySQL or Vitess<br>• Shard key selection<br>• Data redistribution<br>• Cross-shard query handling |
